# L2 Requirements — Amalgam: Local Monolith Runner

## L1-1: Configuration System

### L2-1.1: YAML Configuration Schema
The tool shall use a YAML configuration file (`amalgam.yml`) at the workspace root defining:
- `repositories`: list of repo entries each with `name`, `type` (microservice | library | plugin | dashboard), `path` (local filesystem path), and optional `enabled` (bool, default true)
- `backend.port`: port for the unified host (default 5000)
- `backend.environment`: dictionary of environment variables
- `frontend.dashboardPath`: path to the dashboard repo
- `frontend.port`: port for the frontend dev server (default 3000)

**Traces to**: L1-1
**Acceptance Criteria**:
- A valid `amalgam.yml` can be deserialized into a strongly-typed C# model
- Missing optional fields use documented defaults
- Invalid YAML produces a clear error message with line number

### L2-1.2: Configuration Validation
The tool shall validate configuration on load: all `path` values must point to existing directories, no duplicate `name` values, at most one `dashboard` type entry, and every `microservice` type must contain a `.csproj` file.

**Traces to**: L1-1
**Acceptance Criteria**:
- Validation errors list all problems (not just the first)
- Each error includes the offending field path and value

### L2-1.3: Init Command Generates Scaffold Config
The `init` command shall scan a given root directory for git repos, detect their type by heuristics (presence of `.csproj`, `package.json`, known folder structures), and generate a starter `amalgam.yml`.

**Traces to**: L1-1
**Acceptance Criteria**:
- Running `amalgam init --scan-dir <path>` produces an `amalgam.yml` with discovered repos
- Detected types are reasonable defaults that the user can manually adjust

## L1-2: Local Package Resolution

### L2-2.1: NuGet Local Override
For each `library` repo, the tool shall generate a `Directory.Build.props` or `NuGet.Config` override in each consuming `microservice` repo that redirects the matching NuGet package reference to a local project reference.

**Traces to**: L1-2
**Acceptance Criteria**:
- After running the override, `dotnet build` on a microservice resolves the library from local source, not Nexus
- The original `.csproj` files are not modified; overrides use MSBuild import files

### L2-2.2: npm Local Linking
For each `plugin` repo, the tool shall run `npm link` (or generate workspace-level package resolution) in the `dashboard` project so the plugin resolves to local source.

**Traces to**: L1-2
**Acceptance Criteria**:
- After linking, the dashboard's `node_modules/<plugin-package>` symlinks to the local plugin path
- The dashboard's `package.json` is not permanently modified

## L1-3: Unified Backend Host

### L2-3.1: Dynamic Service Loader
The tool shall create an ASP.NET Core host project (`Amalgam.Host`) that dynamically loads each enabled microservice's `Startup`/`Program` registrations (DI services, middleware, controllers) via reflection or a convention-based interface (`IAmalgamModule`).

**Traces to**: L1-3
**Acceptance Criteria**:
- A microservice implementing `IAmalgamModule` has its `ConfigureServices` and `ConfigureApp` methods invoked at host startup
- Controllers from all loaded microservices are discoverable and routable

### L2-3.2: Route Prefix Isolation
Each microservice's routes shall be mounted under a configurable base path prefix (defaulting to `/<service-name>`) to avoid route collisions.

**Traces to**: L1-3
**Acceptance Criteria**:
- Two microservices both defining `GET /health` are reachable at `GET /<svc1>/health` and `GET /<svc2>/health`
- The prefix is configurable per-service in `amalgam.yml`

### L2-3.3: Shared Service Registration
Shared library DI registrations shall be loaded once and made available to all microservices, avoiding duplicate singleton registrations.

**Traces to**: L1-3
**Acceptance Criteria**:
- A shared library registering a singleton `ICache` is resolved as the same instance across all microservices
- No duplicate service registration warnings

## L1-4: Frontend Plugin Linking

### L2-4.1: Plugin Link Command
The `link-plugins` command shall iterate configured `plugin` repos and run local linking into the dashboard project.

**Traces to**: L1-4
**Acceptance Criteria**:
- Running `amalgam link-plugins` links all enabled plugin repos
- Running with `--only <name>` links a single plugin

### L2-4.2: Plugin Unlink Command
The `unlink-plugins` command shall restore the dashboard to its published-package state.

**Traces to**: L1-4
**Acceptance Criteria**:
- After unlinking, the dashboard resolves plugins from the registry (npm install restores)

## L1-5: CLI Interface

### L2-5.1: CLI Command Structure
The CLI (`amalgam`) shall support these subcommands:
- `init` — generate config
- `validate` — check config
- `build` — build all enabled backend projects
- `run` — start the unified host
- `link-plugins` — link frontend plugins
- `unlink-plugins` — unlink frontend plugins
- `status` — show which services are enabled/discovered

**Traces to**: L1-5
**Acceptance Criteria**:
- `amalgam --help` lists all subcommands with descriptions
- Unknown subcommands produce a helpful error

### L2-5.2: Run Command Options
The `run` command shall support `--port <n>`, `--enable <svc>`, `--disable <svc>`, `--env KEY=VALUE`, and `--verbose` flags to override config at runtime.

**Traces to**: L1-5
**Acceptance Criteria**:
- `--disable SvcA` excludes SvcA even if `enabled: true` in config
- `--env` values are visible to microservice code via `IConfiguration`

## L1-6: Developer Experience

### L2-6.1: Startup Health Summary
On startup, the unified host shall print a table listing each service, its status (loaded/skipped/error), route prefix, and load time.

**Traces to**: L1-6
**Acceptance Criteria**:
- The table is printed to stdout before "Now listening on" message
- Services that failed to load show the error reason

### L2-6.2: Configuration Hot-Reload Toggle
The tool shall support `--watch` mode where changes to `amalgam.yml` (specifically `enabled` flags) trigger a restart of only affected services without full process restart.

**Traces to**: L1-6
**Acceptance Criteria**:
- Toggling a service's `enabled` flag while running in `--watch` mode loads/unloads the service within 5 seconds
- A log message confirms the reload

### L2-6.3: Error Diagnostics
When a microservice fails to load (missing dependency, build error, incompatible interface), the tool shall report the root cause with actionable suggestions.

**Traces to**: L1-6
**Acceptance Criteria**:
- A missing project reference error suggests running `amalgam build` first
- A DI registration conflict identifies the conflicting service type and which modules registered it

## L1-8: Folder Merge / Assembly

### L2-8.1: Merge Configuration Schema
Each repository entry in `amalgam.yml` shall support an optional `merge` property containing:
- `sources`: list of relative folder paths within the repo to merge (e.g., `src/generated/Shared/Types`, `src/static/templates/Shared`)
- `target`: optional relative path to the `.csproj` or `package.json` within the repo that represents the project being assembled (used to locate the project file in the merged output)

When `merge` is present, the tool assembles the listed source folders into a temp staging directory before any build, package resolution, or linking step.

**Traces to**: L1-8
**Acceptance Criteria**:
- A `RepositoryConfig` with `merge.sources` deserializes correctly from YAML
- Missing `merge` property means no merge step (backward compatible)
- An empty `sources` list produces a validation error

### L2-8.2: Folder Merge Service
The tool shall provide a `FolderMergeService` that:
- Creates a temp staging directory at `.amalgam/merged/<repo-name>/`
- Copies all files from each source folder into the staging directory, preserving relative directory structure within each source
- If the repo has a `.csproj` or `package.json` at the `merge.target` path, copies it to the staging root
- Returns the staging directory path so downstream services (NuGet override, module loader, npm link) use it instead of the original `repo.Path`

**Traces to**: L1-8
**Acceptance Criteria**:
- Files from all source folders appear in the merged output directory
- Overlapping filenames from different sources are handled (last source wins, with a warning logged)
- The staging directory is created under `.amalgam/merged/<repo-name>/` relative to the working directory

### L2-8.3: Merge Integrated into Pipeline
The `amalgam build` and `amalgam run` commands shall automatically execute the merge step for any repository with a `merge` configuration before building or loading. The resolved (merged) path shall be used for all downstream operations (NuGet override generation, module loading, npm linking).

**Traces to**: L1-8
**Acceptance Criteria**:
- Running `amalgam build` on a repo with `merge` config merges folders first, then builds from the staging directory
- Running `amalgam run` uses the merged path for module loading
- `amalgam link-plugins` uses the merged path if the plugin has a merge config

### L2-8.4: Merge Clean Command
The `amalgam merge-clean` command shall delete all staging directories under `.amalgam/merged/`.

**Traces to**: L1-8
**Acceptance Criteria**:
- Running `amalgam merge-clean` removes the `.amalgam/merged/` directory
- Running it when no merged directories exist produces no error

### L2-8.5: Merge Validation
The config validator shall validate merge configurations:
- All `sources` paths must exist as directories within the repo path
- The `target` path (if specified) must exist within one of the source folders

**Traces to**: L1-8
**Acceptance Criteria**:
- A nonexistent source path produces a validation error naming the repo and missing path
- Validation errors are collected alongside other config validation errors

## L1-7: .NET Tool Packaging and NuGet Publishing

### L2-7.1: .NET Tool csproj Metadata
The `Amalgam.Cli.csproj` shall include all properties required for .NET global tool packaging:
- `PackAsTool` = true
- `ToolCommandName` = "amalgam"
- `PackageId`, `Version`, `Authors`, `Description`, `PackageReadmeFile`, `PackageLicenseExpression`
- `PackageProjectUrl`, `RepositoryUrl` (placeholder values acceptable)

**Traces to**: L1-7
**Acceptance Criteria**:
- `dotnet pack` on Amalgam.Cli produces a `.nupkg` file in the output directory
- The `.nupkg` contains the tool shim with command name "amalgam"
- The package includes a README and license metadata

### L2-7.2: NuGet Publish Support
The project shall include a CI-friendly publish workflow: `dotnet nuget push` can push the `.nupkg` to nuget.org using an API key.

**Traces to**: L1-7
**Acceptance Criteria**:
- `dotnet pack -c Release` produces a release-mode `.nupkg`
- The package can be pushed to nuget.org via `dotnet nuget push`
- A user can install with `dotnet tool install -g Amalgam.Cli` and then run `amalgam --help`

### L2-7.3: Package Validation
The `.nupkg` shall be valid and installable as a global tool locally before publishing.

**Traces to**: L1-7
**Acceptance Criteria**:
- `dotnet tool install -g --add-source ./nupkg Amalgam.Cli` succeeds from the local pack output
- Running `amalgam --help` after install shows the expected subcommands
- `dotnet tool uninstall -g Amalgam.Cli` cleanly removes the tool

## L1-9: Web Application for Configuration Management

### L2-9.1: Responsive App Shell & Navigation
The app shall have a responsive shell with Angular Material sidenav for navigation. On mobile (≤599px), navigation collapses to a hamburger menu. On tablet (600–1279px), navigation shows as a rail with icons. On desktop (≥1280px), navigation is a full sidebar. Navigation items: Dashboard, Repositories, Backend Config, Frontend Config, Templates, and Settings.

**Traces to**: L1-9
**Acceptance Criteria**:
- Mobile layout shows hamburger menu with bottom navigation bar
- Tablet layout shows icon rail on the left
- Desktop layout shows full sidebar with labels and icons
- All three breakpoints are designed as separate screens

### L2-9.2: Dashboard / Home Screen
The dashboard shall show an overview of the current configuration: total repositories by type (with mat-chips), validation status (pass/fail with error count), quick-action FABs for "New Repository" and "Run Wizard", and recent activity or change history cards.

**Traces to**: L1-9
**Acceptance Criteria**:
- Dashboard displays repository count cards grouped by type (Microservice, Library, Plugin, Dashboard)
- Validation status banner shows pass (green) or fail (red/amber) with error count
- Quick action buttons are visible and accessible
- Layout adapts across mobile, tablet, and desktop

### L2-9.3: Repository List & Management
The app shall display repositories in a responsive list/table with columns for name, type (color-coded chip), path, enabled toggle, and actions (edit, clone, delete). On mobile, the list collapses to cards. Filtering by type and search by name shall be supported.

**Traces to**: L1-9
**Acceptance Criteria**:
- Desktop shows a mat-table with sortable columns
- Mobile shows mat-card list with key info and swipe actions
- Type filter chips and search bar are present
- Enable/disable toggle is inline

### L2-9.4: Repository Add/Edit Form with Validation
The repository form shall use Angular Material form fields with real-time validation. Fields: name (required, unique, alphanumeric+hyphens), type (mat-select), path (with browse/type-ahead), enabled (slide toggle), routePrefix (auto-suggested from name), packageName (conditional on type=Plugin). All validation errors display as mat-error with descriptive messages below each field.

**Traces to**: L1-9
**Acceptance Criteria**:
- Each field shows specific validation error messages (e.g., "Name must be unique", "Path does not exist")
- Type-ahead for path field suggests directory paths
- Route prefix auto-populates as `/<name>` but is editable
- Package name field only appears when type is Plugin
- Form has clear Save and Cancel actions

### L2-9.5: Merge Configuration Section
Within the repository form, when a user adds merge config, a sub-section appears with: a dynamic list of source paths (add/remove with type-ahead), an optional target path field, and validation that sources are not empty. Descriptive help text explains what merge does.

**Traces to**: L1-9
**Acceptance Criteria**:
- "Add Merge Configuration" expansion panel appears within repo form
- Source paths can be added/removed dynamically
- Empty sources list shows validation error
- Help tooltip or info banner explains merge behavior

### L2-9.6: Backend & Frontend Configuration Screens
Separate screens for backend config (port number with validation 1–65535, environment variables as key-value pairs with add/remove) and frontend config (dashboard path with type-ahead, port). Environment variables use a dynamic form array with mat-form-fields.

**Traces to**: L1-9
**Acceptance Criteria**:
- Backend config screen shows port field and environment variable editor
- Port validates range 1–65535 with error message
- Environment variables support add/remove with key-value inputs
- Frontend config shows dashboardPath and port fields

### L2-9.7: Alerts, Dialogs & Notifications
The app shall include designs for: (a) mat-dialog for delete confirmation with warning icon and action buttons, (b) mat-dialog for clone configuration with name input, (c) mat-snackbar notifications for success/error/info, (d) inline alert banners (warning, error, info, success) using mat-card with colored left border and icon.

**Traces to**: L1-9
**Acceptance Criteria**:
- Delete confirmation dialog shows warning icon, item name, and Confirm/Cancel buttons
- Clone dialog shows source name, new name input with validation, and Clone/Cancel buttons
- Snackbar notifications appear at bottom with dismiss action
- Inline alert banners show for validation errors, save success, and informational notices

### L2-9.8: Configuration Wizard (New Config)
A multi-step wizard using mat-stepper shall guide users through creating a new configuration: Step 1 – Scan directory or start blank, Step 2 – Review/edit discovered repositories, Step 3 – Configure backend settings, Step 4 – Configure frontend settings, Step 5 – Review & save. Each step validates before allowing next.

**Traces to**: L1-9
**Acceptance Criteria**:
- Wizard uses horizontal stepper on desktop, vertical on mobile
- Each step shows validation state (complete, error, pending)
- "Scan Directory" step includes path input with type-ahead
- Review step shows full YAML preview
- Back/Next/Finish buttons are clearly visible

### L2-9.9: Templates & Configuration Cloning
The app shall include a Templates screen listing pre-built configuration templates (e.g., "Basic Microservice Setup", "Full Stack with Plugins", "Library Development"). Each template shows a description and "Use Template" button. Configuration cloning shall allow duplicating an existing config with a new name, modifying the clone independently.

**Traces to**: L1-9
**Acceptance Criteria**:
- Templates screen shows template cards with name, description, and repo count preview
- "Use Template" opens the wizard pre-filled with template data
- Clone action available from repository list and config overview
- Clone dialog validates new name uniqueness

### L2-9.10: YAML Preview & Export
The app shall include a YAML preview panel (using a code-style monospace display) that shows the live configuration as valid YAML. Users can toggle between visual editor and YAML view. An export/download button saves the YAML file.

**Traces to**: L1-9
**Acceptance Criteria**:
- YAML preview updates in real-time as configuration changes
- Toggle between visual editor and YAML view is accessible
- Download button exports `amalgam.yml`
- YAML preview uses monospace font with syntax highlighting colors

### L2-9.11: Dark Mode Default Theme
The web application shall use a dark mode color scheme by default across all screens and components. The dark theme shall use a dark background (e.g., #0F172A or similar dark slate), lighter surface colors for cards and containers, light text on dark backgrounds, and adjusted accent/status colors that maintain sufficient contrast (WCAG AA). All reusable components (buttons, inputs, chips, toggles, cards, alerts, dialogs, snackbars, navigation) shall be updated to use dark mode variables. The design variables shall be updated so that all existing screens automatically reflect the dark theme.

**Traces to**: L1-9
**Acceptance Criteria**:
- All screens (mobile, tablet, desktop) render with dark backgrounds and light text
- Cards and surface containers use a slightly lighter dark shade than the background
- Input fields have visible borders and light placeholder/value text on dark backgrounds
- Alert banners maintain distinct color coding (error, success, warning, info) with dark-adjusted tints
- Navigation elements (sidebar, icon rail, bottom nav) use dark surface colors
- Chip colors for repository types remain distinguishable on dark backgrounds
- Toggle switches remain clearly visible in on/off states
- All text meets WCAG AA contrast ratio against its background
